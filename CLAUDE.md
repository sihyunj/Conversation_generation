# CLAUDE.md

Guidance for Claude Code when working in this repository.
**This repo is operated entirely through Claude Code** — there is no human
developer doing manual edits between sessions, so context that would otherwise
live in heads needs to live in this file.

## What this repo is

`Conversation_generation` is a research repo for **synthetic conversation data
generation**, focused on US-English messaging. The pipeline is:

```
1. Personas        →  personas/data/personas.jsonl        (100 hand-curated US personas)
2. Topic pairs     →  personas/data/topics.jsonl          (91 plausible A-B-relationship combos)
3. Seed messages   →  personas/data/seed_messages.jsonl   (66 single-turn style anchors)
4. Conversations   →  conversations/data/conversations.jsonl
                       multi-turn labeled dialogs (trigger yes/no + 9 actions)
                       — generation code shipped; run with API key to populate
5. Eval set        →  (future) hand-written hold-out
```

Inspired by NVIDIA's [Nemotron-Personas-Korea](https://huggingface.co/datasets/nvidia/Nemotron-Personas-Korea).

## Working norms in this repo

- **Branches**: always work on `claude/<short-description>-<id>` branches.
  `main` is for merged work only. Active / recent branches:
  - `claude/create-messaging-personas-1BLeS` — personas module (merged via PR #1).
  - `claude/messaging-conversations-3kLpQ7` — conversations module (current).
- **Per-task work logs**: every meaningful work session adds an entry to
  `WORK_LOG.md`. Newest on top. Include: branch, goal, decisions, tradeoffs,
  validation result, known limitations.
- **Never touch other repos**: GitHub MCP scope is restricted to
  `sihyunj/conversation_generation`.
- **Schemas matter**: any change to `personas/schema.md` MUST be accompanied
  by a matching update in `personas/validate.py` and the prompt templates
  in `personas/prompts/`. Don't drift.
- **Data files are artifacts**: `data/*.jsonl` is regeneratable from
  `_build_seed.py` (deterministic) or `generate_personas.py` + `generate_topics.py`
  (live-API). Don't hand-edit JSONL — edit the source, rebuild.
- **Validation is mandatory**: if you change a persona, topic, or schema,
  re-run `python validate.py` before committing.

## Required tools / environment

- Python 3.11+
- `anthropic` Python SDK (only for live-API regeneration; not needed for
  rebuilding from `_build_seed.py`)
- `tqdm` (for progress bars in live-API scripts)

The shipped data in `personas/data/` was generated by Claude (the assistant)
directly — `_build_seed.py` re-emits the exact JSONL deterministically.

## Running things

```bash
# personas / topics / seed messages
cd personas
python _build_seed.py                                     # deterministic rebuild
export ANTHROPIC_API_KEY=...
python generate_personas.py --count 100 --out data/personas.jsonl   # live regen
python generate_topics.py --in data/personas.jsonl --out data/topics.jsonl --pairs 300
python validate.py

# labeled multi-turn conversations
cd ../conversations
python generate_conversations.py --count 1000 --out data/conversations.jsonl
python validate_conversations.py
python validate_conversations.py --examples              # validates examples/
```

## Conventions

- **Output format**: JSONL throughout (one record per line, UTF-8, no BOM).
- **UUIDs**: real UUID4 in the `uuid` / `topic_id` / `seed_id` field. Stable
  across rebuilds because `_build_seed.py` seeds Python's `random` module.
- **Names**: realistic English / Spanish / Asian / Native American surnames
  reflecting US ethnic distribution. No real-person impersonation; if a
  generated name accidentally matches a real public figure, regenerate.
- **No PII**: synthetic only. Never include real phone numbers, addresses,
  emails, or social handles in any data file.
- **Languages**: narratives are American English. Spanish, Mandarin, Korean,
  Vietnamese, etc. appear in the `native_language` field and may be referenced
  inside narratives, but the canonical narrative text stays English.

## What NOT to do

- Don't add features to `personas/` that the current task doesn't require.
- Don't introduce abstractions, registries, or framework-y patterns. The
  scripts are intentionally flat, top-to-bottom Python.
- Don't add new dependencies without updating `requirements.txt` and a note
  in `WORK_LOG.md`.
- Don't edit `data/*.jsonl` by hand. Edit the source, rebuild, validate, commit.
- Don't push to `main`. Always work on a feature branch.
- Don't create a PR unless explicitly asked.

## When the user asks "what's the state of this repo?"

Read `WORK_LOG.md` (newest entries first) and `personas/README.md`. Those two
files together describe everything. If they don't, update them.
